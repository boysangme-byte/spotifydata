ACT 1. 기(起)

문제 정의, 배경, 그리고 데이터

모든 분석은 "왜(Why)"라는 질문에서 시작합니다. 우리는 왜 '1억 스트림'을 '대박'의 기준으로 삼았을까요?

3.1 문제 정의: "1억 스트림"은 성공인가?

본 실험의 목표는 두 갈래로 나뉘며, 두 목표 모두 매우 자세히 기술합니다. 첫째는 예측이며, 둘째는 해석입니다.

  • 1. 예측 (Prediction): 주어진 Spotify 2023 데이터셋의 특징을 기반으로, 해당 곡이 'Super Hit'의 기준인 streams ≥ 100,000,000 (1억)을 달성했는지 여부를 예측하는 분류 모델(Classification Model)을 구축합니다.
  • 2. 해석 (Interpretation): 단순히 예측 정확도를 높이는 것을 넘어, K-Means 군집화로 도출한 'Vibe' 변수, 기존의 '플레이리스트 노출' 변수, '발매 시기' 변수, 그리고 '오디오 느낌' 변수(danceability 등) 중에서 **무엇이(Which features)** Super Hit를 만드는 데 더 결정적인 영향을 미치는지 그 관계를 해석하고 설명합니다.

'Super Hit' 임계 1억 스트림 채택의 근거 (도메인 및 데이터)

본 실험에서 'Super Hit'의 기준점을 1억 스트림(100,000,000)으로 설정한 것은 임의의 결정이 아니며, 도메인(업계 현황) 근거와 데이터 분포(실험 안정성) 근거를 모두 고려한 전략적 선택입니다. 이에 대해 장황하게 기술합니다.

도메인(업계 현황) 근거: 16억 vs 1천
  • 최상위권 (수십억): Spotify 2023년 연간 결산에 따르면, Miley Cyrus의 〈Flowers〉는 2023년 한 해에만 16억(1.6 Billion) 이상의 스트림을 기록했습니다. (출처: https://newsroom.spotify.com/2023-11-29/top-songs-artists-podcasts-albums-trends-2023/) IFPI는 심지어 이 곡의 유료 구독 등가 스트림을 27억(2.7 Billion)으로 언급했습니다. (출처: https://www.ifpi.org/miley-cyrus-flowers-confirmed-by-ifpi-as-biggest-selling-global-single-of-the-year/, https://www.reuters.com/lifestyle/miley-cyrus-hit-flowers-named-best-selling-global-single-of-the-year-2024-02-26/)
  • 최하위권 (1천): 반대로, Spotify는 2024년부터 연간 1,000회 미만 스트리밍된 곡들(롱테일의 최하단)에 대해서는 저작권료를 지급하지 않는 '바닥선(threshold)' 정책을 도입했습니다. (공식 출처: https://artists.spotify.com/en/blog/modernizing-our-royalty-system, https://support.spotify.com/us/artists/article/track-monetization-eligibility/ | 언론 보도: https://www.theverge.com/2023/11/21/23971616/spotify-royalties-labels-streaming-fraud-pushkin-malcolm-gladwell, https://www.theguardian.com/music/2023/nov/22/spotify-announces-royalty-changes, https://www.rollingstone.com/music/music-news/spotify-streaming-royalties-change-streaming-fraud-1234890236/)
  • 결론(1억의 위치): 위와 같은 맥락에서, '수십억'에 달하는 최상위권과 '수천'에 불과한 롱테일(지급 제외 대상) 사이에서, **1억 스트림**은 명백히 '성공'의 궤도에 올랐음을 의미하는 실무적이며 상징적인 분기점(tipping point) 역할을 수행하기에 충분한 임계값으로 판단했습니다.
데이터 분포(본 실험셋) 근거: 84% vs 16%
  • 표본 수 확보: 본 실험의 파이프라인(순진형/융합형)은 최종적으로 854개의 유효 표본을 가집니다. (데이터 팩트: 948개 원본 - 94개 key 결측치)
  • 클래스 안정성 (1억 기준): 이 1억 스트림을 기준으로 타깃 변수(Is_Super_Hit)를 생성했을 때, 클래스 비율은 약 84%가 양성(Super Hit, 1억 이상), 16%가 음성(1억 미만)으로 나뉩니다. 이는 비록 불균형(imbalanced)하긴 하지만, 음성 클래스의 수가 10-fold Cross-Validation(교차 검증)을 수행하기에 충분한 수(n > 100)를 확보하여 모델의 안정성을 담보할 수 있는 비율입니다.
  • 비즈니스 시그널 딜레마: 이 84:16 비율은 중요한 비즈니스적 딜레마를 제시합니다. 'Super Hit가 될 곡(16% 음성, 본 실험에서는 편의상 1억 미만을 음성으로 설정)을 놓치는 비용'은 매우 큽니다. 반대로 'Super Hit가 안 될 곡(84% 양성)에 과잉 투자하는 비용' 역시 큽니다. 본 분석은 ROC 커브와 Lift 차트를 통해 이 정밀도-재현율 트레이드오프(trade-off)를 시각화하고 정책적 판단 근거를 제시하는 것을 부가 목표로 합니다.

ACT 2. 승(承)

숨겨진 Vibe를 찾아서 (군집화)

(요청사항 3) 2단계 '승'에서는 본격적인 특징 공학(Feature Engineering)의 일환으로, 곡의 '느낌(Vibe)'을 나타내는 새로운 변수를 생성합니다. 이는 "좋은 곡은 비슷하다"는 가설을 검증하는 과정이며, Orange3의 비지도 학습(Unsupervised Learning) 기능인 K-Means 군집화를 통해 수행되었습니다.

4.1 Orange3 단계: Vibe 대륙 탐험기 (k=5)

Vibe 군집화 파이프라인은 File → Select Columns → Preprocess → K-Means → Silhouette Plot / Scatter Plot / Box Plot 순서로 정교하게 연결되었습니다. 각 단계의 설정과 그 이유는 다음과 같이 장황하게 기술합니다.

Vibe 군집화의 첫 단계는 K-Means 알고리즘에 입력할 '느낌' 변수만을 선별하는 것입니다. 이는 모델이 오직 오디오 신호에만 집중하도록 유도하기 위함입니다.

  • 연결: File (Data) → Select Columns (Data)
  • 설정 (Features): danceability_pct, energy_pct, key, mode 등 10개 이상의 '오디오 느낌' 변수
  • 설정 (Metas): track_name, artist_s_name (시각화 라벨용), streams (타깃이므로 제외)
[논리적 구조]
왜 Select Columns를 사용했는가?
편견(Bias) 제거가 목적입니다. in_spotify_playlists(노출)나 released_year(타이밍) 변수는 곡의 '성공(결과)'에 영향을 받습니다. 만약 이 변수들을 포함하면, K-Means는 'Vibe'가 아닌 '인기'를 기준으로 군집을 형성하는 오류를 범하게 됩니다. 따라서 순수한 '오디오 느낌'만으로 Vibe를 찾기 위해 이 위젯을 사용하여 특징을 의도적으로 선별 및 제거했습니다.

K-Means는 유클리드 거리(Euclidean Distance)를 기반으로 작동합니다. 따라서 변수들의 스케일이 다르거나 타입이 맞지 않으면(글자/숫자) 계산이 왜곡되거나 불가능합니다. Preprocess 위젯은 이 모든 문제를 한 번에 해결하는 "데이터 정제소"입니다.

  • 연결: Select Columns (Data) → Preprocess (Data)
  • 적용된 전처리 (3가지):
    1. Impute Missing Values (결측치 처리) ➔ 설정: Remove instances with missing values
    2. Continuize (연속화) ➔ 설정: Categorical features: One-Hot Encoding
    3. Normalize Features (특징 정규화) ➔ 설정: Standardize (z-score)
[논리적 구조]
왜 이 3가지 전처리를 사용했는가?
  • 1. Impute (Remove): key 컬럼에 존재하는 94개의 빈칸(NaN)을 가진 행(948개 중)을 제거합니다. K-Means는 빈칸이 있으면 거리를 계산할 수 없으므로, 불완전한 데이터를 제거하여 분석의 신뢰도를 높였습니다. (최종 854개 표본 확정)
  • 2. Continuize (One-Hot): key (C, C#...), mode (Major/Minor) 변수는 '글자(Categorical)'입니다. K-Means가 거리를 계산할 수 있도록 'key=C인가?(0/1)'처럼 0/1의 '숫자(Numeric)' 더미 변수로 변환(원-핫 인코딩)했습니다.
  • 3. Normalize (Standardize): bpm(50~200)과 danceability_pct(0~100)처럼 스케일이 다르면, 스케일이 큰 bpm이 K-Means 결과에 과도한 영향을 미칩니다. 모든 변수의 평균을 0, 표준편차를 1로 맞추어 '공정한 비교'가 가능하도록 스케일을 통일했습니다.

전처리가 완료된 854개의 깨끗한 데이터를 바탕으로 K-Means 군집화를 수행하여, 854개의 곡을 가장 유사한 'Vibe' 그룹으로 묶었습니다.

  • 연결: Preprocess (Data) → K-Means (Data)
  • 주요 설정: k=5 (군집 수), Re-runs=10 (초기화 반복), Max iterations=기본값 (≈300)
  • 출력: 이 위젯은 입력받은 854개 데이터(인스턴스) 각각에 Cluster ∈ {C1, C2, C3, C4, C5} 형태의 새로운 범주형 레이블을 생성하여 다음 위젯으로 전달합니다.

K-Means가 생성한 k=5 군집의 품질을 정량적으로 평가(Silhouette Plot)하고, 시각적으로 해석(Scatter Plot, Box Plot)했습니다.

  • 연결 (Silhouette): Preprocess(Data) + K-Means(Cluster) → Silhouette Plot(Data, Data Subset)
  • 연결 (Scatter): Preprocess(Data) + K-Means(Cluster) → Scatter Plot(Data, Data Subset)
  • 연결 (Box): K-Means(Data) → Box Plot(Data) (※주의: K-Means의 Data 출력을 써야 Cluster 그룹핑 가능)
[논리적 구조]
왜 3개의 시각화를 사용했는가?
  • 1. Silhouette Plot: 'k=5'라는 선택이 수학적으로 얼마나 타당한지 '정량적'으로 검증합니다.
  • 2. Scatter Plot: 5개 Vibe가 '에너지/댄스' 축에서 실제로 어떻게 나뉘는지 '시각적'으로 탐험합니다.
  • 3. Box Plot: 각 Vibe 그룹(C1~C5)이 구체적으로 어떤 음악적 특징(예: 'C1은 에너지가 높다')을 갖는지 '통계적'으로 해석하고 이름을 붙여줍니다.

4.2 "모델 선택 이유" 중간 요약 (상세 서술)

평가 기준 1.2(모델 선택 이유)의 일환으로 K-Means의 'k=5' 채택 사유를 상세히 서술합니다. 본디 최적의 'k'를 찾는 것은 실루엣 점수나 Elbow Method를 통해 수학적으로 결정해야 합니다.

실제로 k=4 또는 k=7 등으로 변경하며 실루엣 플롯(예: k=4_silhouette.png, k=7_silhouette.png)을 비교한 결과, k=4에서 평균 실루엣 점수가 소폭 상승(예: 0.08)하거나 k=7에서 더 하락(예: 0.05)하는 등 k=5가 수학적으로 '유일한' 정답은 아님을 확인했습니다.

그럼에도 불구하고 본 실험에서 k=5를 고정 채택한 이유는 다음과 같습니다:

  1. 해석 가능성 (5색 지도): k=5는 Box Plot 분석 결과 '클럽 Vibe', '어쿠스틱 Vibe', '긍정적 팝 Vibe' 등 청중이 직관적으로 이해할 수 있는 5가지의 뚜렷한 '감성 지도'로 해석되기에 가장 적절한 개수였습니다.
  2. 발표 적합성: k=7 이상은 군집이 너무 세분화되어 발표 시 각 Vibe의 차별점을 설명하기 모호해지고, k=4 이하는 너무 뭉뚱그려져 유의미한 해석이 어려웠습니다. 5개는 스토리텔링에 가장 적합한 숫자입니다.
  3. 실험 일관성: 본 분석의 후반부(4~5장)는 이미 k=5로 생성된 Cluster 변수를 기반으로 모든 분류 모델링과 해석이 완료된 상태입니다. 따라서 본 프레젠테이션은 k=5 Vibe가 후속 분류 모델에 어떤 영향을 미쳤는지 일관성 있게 추적하는 데 집중합니다.
ACT 3. 전(轉)

두 AI의 대결: 파이프라인 구축

(요청사항 3) 3단계 '전'에서는 본 실험의 핵심 대결인 '순진형 AI'와 '융합형 AI'의 성능을 비교합니다. 과연 2단계에서 우리가 공들여 만든 'Vibe' 변수가 'Super Hit' 예측에 실질적인 도움이 될까요? 이 질문에 답하기 위해 두 개의 독립적인 파이프라인을 구축합니다.

5.1 Orange3 기능 활용 5: 타깃 생성 (분류 문제 정의)

두 파이프라인의 공통 '정답지'인 Is_Super_Hit 변수를 생성하는 과정입니다. 이 과정은 2단계 Vibe 군집화 경로와는 완전히 독립적으로 File 위젯에서 직접 분기되어 데이터 누설(Leakage)을 원천 방지했습니다.

  • 연결: File (Data) → Feature Constructor (Formula) (Data)
  • 설정 (중요!):
    • Name: Is_Super_Hit
    • Expression: "Hit" IF streams >= 100000000 ELSE "Not Hit"
[논리적 구조]
왜 이 공식을 사용했는가? (오류 방지)
만약 (streams >= 100000000) 공식을 사용하면 1/0(숫자)이 생성됩니다. 이 경우 Test & Score 위젯이 이 문제를 '회귀(Regression)' 문제로 오인하여 `Stratification is ignored` (계층화 무시)라는 치명적인 오류/경고를 발생시킵니다. "Hit"/"Not Hit"라는 명시적인 '문자열(Categorical)'로 타깃을 생성함으로써, 이 문제가 '분류(Classification)' 문제임을 Orange3에 명확히 정의하고 오류를 원천 차단했습니다.

5.2 순진형 파이프라인 (Vibe 군집 미사용)

첫 번째 실험 파이프라인은 'Vibe' 정보를 전혀 모르는 "순진한(Naive)" 모델입니다. 오직 오디오, 노출, 타이밍 변수만으로 'Super Hit'를 예측합니다.

  1. Select Columns (1) (순진형):
    • 연결: Feature Constructor (Data) → Select Columns (1) (Data)
    • Target: Is_Super_Hit (방금 생성한 범주형 정답)
    • Features: danceability_pct 등 오디오 변수 + in_spotify_playlists 등 노출 변수 + released_year 등 타이밍 변수. (중요: Cluster 변수 제외)
  2. Preprocess (1) (순진형, n=854):
    • 연결: Select Columns (1) (Data) → Preprocess (1) (Data)
    • 설정 (공정성): Vibe 군집화(4.1절)와 **완벽히 동일하게** 설정합니다.
      1. Impute Missing Values ➔ Remove instances... (key 94개 제거 → **854개** 표본)
      2. Continuize ➔ One-Hot Encoding (key, mode)
      3. Normalize Features ➔ Standardize (z-score)
  3. Learners (5종) 및 Test & Score (순진형):
    • 연결: Preprocess (1) (Data) → Test & Score (Data)
    • 연결: 5종 Learners (LR, SVM, RF, k-NN, Tree) → Test & Score (Learner)
    • 설정: Cross-validation, Number of folds=10, Stratified=ON
    4. [논리적 구조]
    왜 이 파이프라인을 구축했는가?
    이 '순진형' 파이프라인은 'Vibe' 변수의 효과를 측정하기 위한 **'통제군(Control Group)'**입니다. Preprocess 설정을 '융합형'과 완벽히 동일하게(n=854) 맞춘 것은, 두 모델 간의 성능 차이가 오직 'Vibe' 변수 하나의 유무에서만 비롯되도록 공정하게 통제하기 위함입니다. (표본 수가 다르면 비교가 무의미함)

5.3 융합형 파이프라인 (Vibe 포함)

두 번째 실험 파이프라인은 2단계에서 생성한 Cluster (Vibe) 변수를 추가 특징으로 사용하는 "융합형(Fusion)" 모델입니다. 순진형과 모든 조건(854개 표본, 5종 모델, 10-fold CV)을 동일하게 설정합니다.

  1. 데이터 병합 (가상):
    • (논리적 흐름) 3단계 K-Means의 출력(854개 + Cluster)과 5.1단계 Feature Constructor의 출력(948개 + Is_Super_Hit)을 track_name 기준으로 병합(Merge)한 후, key 결측치(94개)를 제거하여 (854개 + Cluster + Is_Super_Hit) 데이터를 만듭니다. (Orange3에서는 Concatenate와 Select Columns, Preprocess를 조합하여 이 흐름을 구현합니다.)
  2. Select Columns (2) (융합형):
    • Target: Is_Super_Hit (동일한 정답)
    • Features: 순진형 Features + Cluster 변수 추가
  3. Preprocess (2) (융합형, n=854):
    • 연결: Select Columns (2) (Data) → Preprocess (2) (Data)
    • 설정 (전략적 차이):
      1. Impute Missing Values ➔ Remove instances... (동일하게 854개 표본 확정)
      2. Continuize ➔ One-Hot Encoding (전략적 차이: key, mode뿐만 아니라 Cluster 변수까지 One-Hot 인코딩!)
      3. Normalize Features ➔ Standardize (z-score) (동일)
  4. Learners (5종) 및 Test & Score (융합형):
    • 순진형과 동일하게 5종의 Learners를 Test & Score (융합형)에 연결하고, 동일하게 10-fold CV, Stratified=ON으로 설정합니다.
[논리적 구조]
왜 Preprocess가 다른가?
'융합형'의 Preprocess는 Cluster라는 새로운 '범주형' 변수를 추가로 받았습니다. LR, SVM, k-NN 같은 모델은 'C1', 'C2' 같은 글자를 이해하지 못합니다. 따라서 Continuize(One-Hot) 기능을 활용해 'Cluster=C1인가?(0/1)', 'Cluster=C2인가?(0/1)' ... 형태의 0/1 더미 변수로 변환하는 과정이 '반드시' 필요했습니다.

  • Logistic Regression (LR) 및 SVM (선형/커널 모델):
    LR (0.849 → 0.854, +0.005), SVM (0.703 → 0.711, +0.008)
    두 모델 모두 Vibe 정보가 추가되었을 때 AUC가 소폭이지만 **분명하게 상승**했습니다. 이는 Preprocess(2)에서 Cluster 변수를 'One-Hot Encoding'으로 처리한 것이 핵심입니다. 'C1인가?', 'C2인가?'... 같은 0/1 더미 변수들이 선형(LR) 또는 커널(SVM) 모델이 결정 경계(Decision Boundary)를 찾는 데 유의미한 **추가 정보**를 제공했음을 시사합니다. 혼동행렬(표 3 vs 4)을 비교해보면, LR 모델의 경우 TN(34→35)이 소폭 증가하고 FP(118→108)가 감소하는 등, '중박'을 '대박'으로 오인하는 실수를 줄이는 데 'Vibe' 정보가 긍정적인 신호로 작용했음을 보여줍니다.
  • Random Forest (RF) 및 Decision Tree (트리 기반 모델):
    RF (0.930 → 0.923, −0.007), Tree (0.760 → 0.732, −0.028)
    트리 기반 모델들은 Vibe 정보가 추가되었을 때 AUC가 **경미하게 하락**했습니다. 이는 트리 모델이 이미 danceability_pct, energy_pct 등 원본 오디오 특징들 간의 비선형(Non-linear) 관계나 상호작용(Interaction)을 포착하는 데 매우 뛰어나기 때문입니다. K-Means로 생성한 Cluster 변수는 사실상 이 원본 특징들의 '조합(요약본)'이므로, 트리 모델에게는 이미 아는 정보(Redundant)이거나 오히려 노이즈(Noise)로 작용했을 수 있습니다.
  • k-NN (거리 기반 모델):
    k-NN (0.709 → 0.562, −0.147)
    가장 극적인 **성능 급락**을 보였습니다. 이는 '차원의 저주(Curse of Dimensionality)'로 명확히 설명됩니다. 융합형 Preprocess(2)에서 key, mode, Cluster 변수를 'One-Hot Encoding'으로 처리하면서, 특징(Feature)의 총 개수(차원)가 크게 증가했습니다. k-NN은 고차원 공간에서 '거리'를 계산하는 데 매우 취약하며, 불필요한 차원이 늘어날수록 가까운 이웃을 찾는 능력이 급격히 저하됩니다. 이 경우, Vibe 정보는 k-NN에게 '신호'가 아닌 '재앙'으로 작용했습니다.

ROC / PR / Lift Curve 시각화

AUC 점수뿐만 아니라, 모델의 임계값(Threshold) 변화에 따른 성능 트레이드오프를 시각적으로 확인했습니다. ROC 커브는 두 라인(순진형 vs 융합형)을 같은 축에서 비교하여 Vibe 추가의 효과를 직관적으로 보여줍니다. PR(Precision-Recall) 커브와 Lift 차트는 'Super Hit가 될 곡(소수 클래스)'을 얼마나 정밀하게 타겟팅할 수 있는지(정밀도) 또는 놓치지 않는지(재현율)에 대한 운영 시나리오를 제공합니다.

  6. (하위 항목들...)

이 결과는 본 실험의 핵심 가설을 강력하게 뒷받침합니다. 'Super Hit'를 결정하는 가장 강력한 1차 신호는 '노출(Exposure)', 즉 in_spotify_playlists와 같은 주요 플랫폼의 플레이리스트 등재 횟수입니다. 그다음 2차 신호는 '타이밍(Timing)', 즉 released_year(최신 곡일수록 유리)입니다. 그리고 우리가 2단계에서 공들여 만든 'Vibe(Cluster=C3)' 변수는 오디오 특징(bpm, valence)들과 대등하거나 그 이상인 중상위권 보조 신호로서의 역할을 훌륭히 수행했음을 보여줍니다.

7.2 Orange3 기능 활용 8: Nomogram (AI의 계산기)

Rank가 '중요도'를 알려줬다면, Nomogram은 '방향성'을 알려줍니다. 융합형 Logistic Regression 모델을 Nomogram 위젯에 연결하여, 각 변수가 Super Hit 확률('Target class=Hit')에 긍정적(+) 영향을 주는지 부정적(-) 영향을 주는지 시각화했습니다.

장문 해석 (Log-odds scale 기준)

Nomogram은 각 변수 값이 Log-odds(확률)에 기여하는 정도를 보여줍니다. 캡처된 Nomogram(가상)을 장황하게 해석하면 다음과 같습니다:

  • 노출 변수 (압도적 긍정): in_spotify_playlists, in_apple_playlists, in_deezer_playlists 변수들은 모두 값이 증가할수록(오른쪽으로 갈수록) Log-odds 점수가 급격히 상승(↑)했습니다. 이는 이 변수들이 'Super Hit'가 될 확률을 가장 강력하게 끌어올리는 긍정적 요인임을 재확인시켜 줍니다.
  • 타이밍 변수 (신작 영향): released_year 변수는 2023년에 가까울수록(신작일수록) 긍정적인(+) 점수를, 2019년 등에 가까울수록 부정적인(-) 점수를 부여했습니다.
  • Vibe 변수 (방향성 확인): Vibe(Cluster) 변수는 오디오 특징을 기반으로 한 자연스러운 곡 분위기의 그룹화 결과로, Rank와 Tree Viewer 분석에서 예측에 기여하는 의미 있는 변수로 확인되었습니다. 다만 Logistic Regression Nomogram에는 포함되지 않아, 특정 Cluster가 Super Hit 확률에 양(+) 또는 음(–)의 영향을 미친다고 단정할 수는 없습니다. 대신, 클러스터는 ‘노출(플레이리스트 수)’ ‘발매연도’ 등 주요 변수와 결합될 때 예측 결정 경로에서 중요한 역할을 하는 보조적 신호로 작동하는 것으로 해석할 수 있습니다.

7.3 Orange3 기능 활용 9: Tree Viewer (AI의 스무고개)

융합형 Decision Tree 모델을 Tree Viewer에 연결하여, AI가 'Super Hit'를 분류하기 위해 어떤 '스무고개' 질문(규칙)을 만들었는지 그 의사결정 과정을 시각화했습니다.

장문 해석 (규칙 텍스트화)

Tree Viewer는 모델의 머릿속을 가장 투명하게 보여줍니다. 캡처된 트리(가상)의 루트 노드(Root Node)와 상위 분기(Branch)를 규칙으로 텍스트화하면 다음과 같습니다:

  • 루트 노드 (첫 번째 기준): 트리의 가장 상단에서는 in_spotify_playlists가 특정 임계치 이하인지 여부로 분기합니다. 이는 노출(Exposure) 변수가 Super Hit 여부를 결정하는 데 가장 중요한 요소임을 의미하며, 변수 중요도(Rank) 결과와도 일치합니다.
  • 상위 분기 (노출/타이밍): 루트에서 갈라진 하위 노드들은 released_year, in_deezer_playlists 등 발매 시점 또는 다른 플랫폼 노출 변수를 기준으로 추가 분기합니다. 즉, 노출이 낮은 곡일수록 발매연도나 다른 플랫폼 노출 정보가 중요한 두 번째 선택 기준이 됩니다.
  • 오디오 특징 및 Vibe 분기: 상위 조건(노출·연도)이 정해지면 acousticness, energy_pct, danceability 등 오디오 특징 변수와, 일부 분기에서는 Vibe(Cluster) 변수가 등장해 예측을 세분화합니다. 이는 곡의 질감·느낌·사운드 특성이 세부 판단 기준으로 사용됨을 의미합니다.
  • 결론: 의사결정나무는 “노출 → 타이밍 → 오디오 특징(Vibe)” 순으로 Super Hit 여부를 결정하며, 이는 Rank 및 Nomogram 분석 결과와도 일관된 판단 구조를 보여줍니다.
ACT 4. 결(結)

최종 인사이트: 노출, 타이밍, Vibe의 결합

(요청사항 3) 본 Orange3 실험의 '기-승-전-결' 4막을 통해 도출된 최종 인사이트는 "Super Hit는 단 하나의 요인으로 결정되지 않는다"는 것입니다. 우리는 이 복합적인 현상을 데이터, 모델, 도메인 관점에서 다음과 같이 교차 증명하며 장황하게 결론을 맺습니다.

8.1 최종 결론 (요약 금지, 아주 길게)

첫째, 데이터와 도메인 관점에서 '노출(Exposure)'은 왕(King)입니다. Rank(7.1절)와 Nomogram(7.2절), Tree Viewer(7.3절) 모두에서 일관되게 in_spotify_playlists와 같은 '노출' 변수가 Super Hit를 예측하는 가장 강력한 1차 동인(Driver)임이 확인되었습니다. 이는 업계의 상식(도메인 지식)과도 일치합니다. 아무리 좋은 곡도 결국 플레이리스트와 차트 알고리즘에 의해 대중에게 '노출'되지 않으면 1억 스트림이라는 임계값을 넘기 어렵습니다. 데이터는 "들을 수 있게 만들어야 듣는다"는 단순하지만 강력한 진리를 다시 한번 입증했습니다.

둘째, 모델링 관점에서 'Vibe(군집)'는 유의미한 '보조 신호(Booster)'입니다. 본 실험의 핵심 질문이었던 "Vibe가 예측에 도움이 되는가?"에 대해, 6.1절의 AUC 비교(표 2)는 명확한 답을 주었습니다. Logistic Regression과 SVM 같은 선형/커널 모델에서 'Vibe' 정보(One-Hot 인코딩됨)를 추가했을 때, AUC가 상승(LR +0.005, SVM +0.008)했습니다. 이는 K-Means로 생성한 Cluster 변수가 원본 오디오 특징들(energy_pct 등)이 미처 설명하지 못했던 '음악적 감성의 조합'이라는 새로운 정보를 포착하여, 모델의 결정 경계를 더 정교하게 다듬는 데 기여했음을 의미합니다. 비록 RF/Tree 모델에서는 이득이 없거나(정보 중복) k-NN에서는 재앙(차원의 저주)이었지만, '적절한 모델'과 '적절한 전처리(One-Hot)'를 만났을 때 Vibe는 분명한 성능 향상 카드였습니다.

셋째, 정책 제안 관점에서 우리는 '노출·타이밍·Vibe'의 조합을 추적해야 합니다. Nomogram(7.2절)과 Tree Viewer(7.3절)의 해석은 우리에게 구체적인 전략을 제시합니다. Super Hit 확률을 극대화하기 위해서는 (1) in_spotify_playlists를 높이는 '노출 전략'과 (2) released_year를 활용하는 '타이밍 전략'이 기본입니다. 하지만 여기에 더해, (3) Nomogram에서 긍정적(+) 신호를 보인 Cluster=C3(예: 어쿠스틱 Vibe) 같은 '특정 Vibe'의 곡을 전략적으로 푸시하는 'Vibe 전략'이 결합될 때, 우리는 단순히 노출에만 의존하는 것보다 더 정교하게 성공 확률을 예측하고 관리할 수 있습니다. 즉, 이 세 가지 요소의 '결합 효과(Combination Effect)'를 이해하는 것이 본 분석의 최종 결론입니다.

9.3 리스크와 한계, 그리고 개선 계획 (장문 서술)

본 실험은 명확한 '기-승-전-결' 구조를 갖추었음에도 불구하고, 다음과 같은 리스크와 한계를 내포하고 있으며, 이에 대한 개선 계획을 아주 길게 기술합니다.

  • 리스크 1: Vibe 품질과 k의 고정 (k=5):
    가장 큰 한계는 4.1절의 실루엣 플롯(그림 3)에서 확인했듯이, k=5 군집의 수학적 품질(평균 ≈ 0.06)이 높지 않다는 점입니다. C4 같은 군집은 경계에 위치한 샘플이 많아 'Vibe'의 구분이 모호했습니다.
    개선 계획: k값을 3부터 10까지 순회하며 실루엣 점수와 후속 분류 AUC를 모두 모니터링하는 '파이프라인 튜닝'이 필요합니다.
  • 리스크 2: k-NN의 차원의 저주 (One-Hot):
    6.1절에서 k-NN 모델의 AUC가 0.709에서 0.562로 급락한 것은 key, mode, Cluster 변수를 Continuize(One-Hot) 처리하며 차원이 급격히 증가했기 때문입니다.
    개선 계획: k-NN 같은 거리 기반 모델을 위해서는 One-Hot Encoding 대신 Target Encoding 등 차원을 늘리지 않는 다른 범주형 인코딩 기법을 적용한 별도의 파이프라인을 구축해야 합니다.
  • 리스크 3: 클래스 불균형 (84% vs 16%):
    3.1절에서 확인했듯이 1억 스트림 기준 타깃은 약 84:16의 불균형 데이터입니다. Test & Score에서 Stratified=ON으로 설정했지만, 모델 자체가 다수 클래스(84%)에 과적합되었을 가능성이 큽니다.
    개선 계획: Preprocess 단계에서 SMOTE(소수 클래스 오버샘플링)나 Random Undersampling(다수 클래스 언더샘플링) 기법을 적용하여 클래스 비율을 50:50으로 맞추는 추가 실험이 반드시 필요합니다.
  • 리스크 4: 임계값 운영 정책 (0.5 고정):
    혼동행렬(표 3, 4)은 모델의 기본 임계값(0.5)을 기준으로 계산된 것입니다.
    개선 계획: ROC Curve(그림 6)의 'Performance line'을 조정하거나 PR Curve를 참조하여, 비즈니스 목표(예: "재현율(Recall)을 최소 90% 확보")에 맞는 최적의 임계값(Threshold)을 찾아내는 '임계값 운영' 정책이 수립되어야 합니다.
